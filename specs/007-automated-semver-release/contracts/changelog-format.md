# Contract: CHANGELOG.md Format

**Purpose**: Define the changelog file format at the repository root for tracking releases and changes.

**Applies to**: `CHANGELOG.md` (repository root)

## Format

```markdown
# Changelog

All notable changes to anvil are documented here.
Format follows [Conventional Commits](https://www.conventionalcommits.org/) section labels.

Sections: `feat`, `fix`, `perf`, `refactor`, `chore`, `docs`, `ci`, `test`, `style`, `build`

Version levels:
- **MAJOR** (X.0.0): Breaking changes
- **MINOR** (x.Y.0): New features
- **PATCH** (x.y.Z): Bug fixes and hot fixes

---

## v0.1.1 (2026-06-14)

### Fix

- **api**: correct tokenizer overflow in inference endpoint

---

## v0.1.0 (2026-06-10)

### Feat

- Initial release with training pipeline and web dashboard
```

## Rules

1. **Reverse chronological**: Newest entry at the top
2. **Section labels**: Use type labels (`Feat`, `Fix`, `Perf`, etc.) — not raw conventional commit types
3. **Scope prefix**: Each entry SHOULD include scope in bold: `-**scope**: description`
4. **Version headers**: `## vX.Y.Z (YYYY-MM-DD)`
5. **Separator**: Each version entry separated by `---`
6. **No empty sections**: Omit sections with no entries
7. **Generated by commitizen**: `cz bump --changelog` appends entries automatically
8. **PR descriptions**: Additional context from PR description body may follow the auto-generated entry

## Validation

```bash
# Verify changelog has entries
head -5 CHANGELOG.md

# Verify most recent version entry
grep "^## v" CHANGELOG.md | head -1
```
