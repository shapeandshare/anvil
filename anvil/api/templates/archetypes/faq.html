Frequently Asked Questions

Everything you’ve wanted to ask about how this tiny transformer works, what it can do, and why it behaves the way it does.

Common questions about how anvil works and what it can (and cannot) do.

Does the model actually understand anything? [+]

No. The model has no understanding, consciousness, or intent. It is a statistical pattern matcher: given a sequence of characters, it predicts the next most likely character based on patterns in its training data. When it generates something that looks meaningful, it is reproducing statistical regularities, not thinking. This is sometimes called a "stochastic parrot."

Why does it hallucinate? [+]

Hallucination is just the model generating plausible-looking text that happens to be factually wrong. Since it has no understanding of truth, it simply produces whatever sequence of characters is statistically likely. If the training data contained patterns like "Einstein was born in 1879," the model may reproduce that. But it can just as easily generate "Einstein was born in 1923" if that fits a statistical pattern. Every output is just a grammar exercise, not a factual statement.

How is this related to ChatGPT? [+]

This is the same fundamental architecture: a decoder-only transformer with self-attention, trained to predict the next token. The differences are scale: ChatGPT has billions of parameters, was trained on trillions of tokens, and used thousands of GPUs for weeks. anvil has 4,192+ parameters (depending on architecture config), trains on kilobytes of text in seconds, and fits on a single CPU or laptop GPU. Every concept here scales up directly to production models — just with 10,000,000× more data and compute.

What architecture does anvil use? [+]

anvil uses the Llama architecture: a decoder-only transformer with Rotary Position Embedding (RoPE), SwiGLU gated MLP, RMSNorm with learned scale parameters, and bias-free linear layers. It does NOT use learned position embeddings (WPE), ReLU activations, or LayerNorm — those were removed in the Llama migration (see ADR-007). The architecture is identical whether training on CPU (pure Python) or GPU (PyTorch). See the Parameters walkthrough for a full breakdown of every weight matrix.

Can I export and use the model elsewhere? [+]

Yes. After training, a trained model can be exported to safetensors — the standard format used by HuggingFace transformers. The export generates model.safetensors (weights with HF-convention tensor names), config.json (LlamaConfig-compatible), and tokenizer.json (character-level tokenizer metadata). These can be loaded by any Llama-compatible tool: HuggingFace, llama.cpp, vLLM, or custom PyTorch code. See the Model Export walkthrough for details on the tensor name mapping and config generation.

Can I train on GPU, or only CPU? [+]

All three. anvil has three training backends with identical Llama architecture. The CPU backend is implemented in pure Python (stdlib only, no framework) with a custom Value autograd graph — this is the default. The GPU backend uses PyTorch tensors and runs 10-100x faster on CUDA (NVIDIA) or MPS (Apple Silicon). The Cloud backend dispatches to Modal for training on remote GPUs, with artifacts synced back to MLflow. Switching is automatic: TrainingService detects the best available local device (CUDA > MPS > CPU) or can be set to Modal for remote execution. GPU-trained weights are bridged back into a CPU LlamaModel so downstream code never knows which backend ran. Check the Operations page to see detected GPU info, or the Cloud Compute walkthrough for remote setup.

How does training data flow through the system? [+]

When you click "Start Training," the browser POSTs a config to FastAPI, which delegates to TrainingService. The service loads documents from the database or filesystem in a thread pool, resolves the compute device, then dispatches to the core engine (CPU or GPU). The engine calls a progress callback every step, which uses asyncio.run_coroutine_threadsafe() to push metrics into an SSE queue. The browser receives real-time events: loss, steps/sec, ETA, and optimizer state. After training completes, the model is logged to MLflow, saved to disk, and optionally exported to safetensors. See the Data Flow walkthrough for the full pipeline diagram.

What MLflow integration exists? [+]

Every training run is automatically tracked via MLflow. Hyperparameters (n_embd, n_head, n_layer, learning rate, etc.) are logged as params. Final loss, device, and elapsed time are logged as metrics. Model artifacts (model.json, samples.txt, and safetensors exports) are uploaded to the MLflow artifact store. The model is auto-registered in the MLflow Model Registry under a name derived from the training data source (dataset or corpus), grouping all versions trained from the same data. You can open the MLflow UI directly from the operations page or experiments page.

Why does it say "8 interactive lessons" but there are more? [+]

The learning arc grows as new content is added. The Learning Index page dynamically shows all available lessons — check it for the full up-to-date list. As of the latest release the arc spans 14 interactive walkthroughs plus the FAQ and Glossary, including step-by-step deep dives into data flow, model export, the forward pass computation graph, full architecture visualization, and cloud compute with Modal. The arc is defined in LEARNING_ARC in anvil/api/v1/learning.py and automatically wired with prev/next navigation.

Where are the progressive code examples (train0–train5)? [+]

The progressive walkthroughs live in examples/train0.py through examples/train5.py. They progress from bigram counting (no neural net) through manual backprop, RMSNorm, RoPE attention, and full Llama blocks to a complete multi-layer model with safetensors export. Each script is self-contained and runnable. See the Progressive Walkthroughs reference in the vault for a detailed breakdown of what each script teaches and the pedagogical design behind the progression.

Is there a glossary of terms? [+]

Yes. The documentation vault includes a Glossary with definitions for every technical term: Value, RMSNorm, KV Cache, AdamW, BOS, Softmax, Cross-Entropy, State Dict, RoPE, SwiGLU, Safetensors, Dual Backend, and more. It lives at docs/vault/Reference/Glossary.md and is accessible from the learning page.

Why is it so slow? [+]

The CPU backend is implemented in pure Python (stdlib only, no framework) — no GPU acceleration, no vectorised ops. Every forward pass loops through scalar operations (add, multiply, softmax) one value at a time. Production models use GPU kernels that process thousands of operations in parallel. The slowness is intentional: it lets you inspect every intermediate value and watch the computation unfold step by step — something impossible with optimised frameworks like PyTorch. If you have a GPU, anvil can use a PyTorch backend that runs 10-100x faster — see “Can I train on GPU, or only CPU?” above.

Can I make it generate better names? [+]

Better names require better data or more training. The default dataset is a small collection of medieval guild names. You can upload your own .txt dataset through the Data tab and retrain. Training for more steps, increasing n_embd, or adding n_layer also helps the model pick up longer-range patterns. But with 4,192+ parameters (depending on architecture config), it will never produce genuinely creative or coherent names — it can only recombine patterns it has seen.

What if I change the dataset? [+]

The model learns whatever character-level patterns exist in the training text. Give it Shakespeare and it will generate Shakespeare-like character sequences. Give it Python code and it will generate plausible-looking code (though with only 4,192+ parameters (depending on architecture config) it won't compile). The vocabulary is built from the unique characters in the dataset, so a dataset with more diverse characters (uppercase, digits, punctuation) gives the model more tokens to work with but spreads its limited capacity thinner.
